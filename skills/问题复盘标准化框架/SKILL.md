# 问题复盘标准化框架（RCA+）

**创建时间**：2026-04-08 19:43
**方法论来源**：Root Cause Analysis (RCA) + 持续改进
**适用场景**：任何工作失误、流程缺陷、系统问题的复盘和改进

---

## 📋 框架概述

**RCA+** = Root Cause Analysis（根本原因分析）+ Action（行动固化）+ Prevention（预防机制）

**核心流程**：
```
问题发生 → 描述问题 → 分析原因 → 设计方案 → 固化流程 → 预防复发
   ↓           ↓           ↓           ↓           ↓           ↓
 事件      What        Why        How        Standard     Check
```

## 🎯 第二阶段：根本原因分析（Why）

### 2.1 5 Why 分析法

**连续追问 5 次"为什么"**：

```
问题：[初始问题]

第 1 问：为什么会发生？
答：[直接原因]

第 2 问：为什么会有这个直接原因？
答：[间接原因]

第 3 问：为什么会有这个间接原因？
答：[深层原因]

第 4 问：为什么会有这个深层原因？
答：[系统原因]

第 5 问：为什么系统允许这个问题存在？
答：[根本原因]
```

**示例（周报遗漏美术组）**：
```
问题：周报汇总遗漏美术组工作内容

第 1 问：为什么会遗漏？
答：汇总时没有审查美术组成员的日报

第 2 问：为什么没有审查美术组日报？
答：没有按岗位逐一审查的流程

第 3 问：为什么没有这个流程？
答：依赖印象式汇总，而非清单式汇总

第 4 问：为什么依赖印象而非清单？
答：没有建立标准化工作流程

第 5 问：为什么没有标准化流程？
答：缺乏问题复盘和流程固化机制
```

## 🚨 第五阶段半：异常处理机制（QA 部门 2026-04-08 新增）

### 5.9 异常分类与处理流程

#### 异常类型 1：数据读取异常

| 项目 | 内容 |
|------|------|
| **现象** | truncated=true 持续多次 / API 报错 |
| **处理流程** | a. 重试机制（最多 3 次，间隔 5 秒）<br>b. 降级方案（手动导出 CSV）<br>c. 上报机制（通知负责人） |
| **记录要求** | 记录异常原因、处理过程、结果 |

#### 异常类型 2：数据不一致

| 项目 | 内容 |
|------|------|
| **现象** | 汇总数据与原始数据不符 |
| **处理流程** | a. 立即停止写入<br>b. 重新读取验证<br>c. 人工核对关键数据<br>d. 确认无误后继续 |
| **记录要求** | 记录不一致内容、原因分析、处理结果 |

#### 异常类型 3：流程中断

| 项目 | 内容 |
|------|------|
| **现象** | 中途中断（系统故障/人员离开） |
| **处理流程** | a. 断点记录（记录已完成步骤）<br>b. 恢复机制（从断点继续）<br>c. 验证机制（恢复后验证完整性） |
| **记录要求** | 记录中断原因、断点位置、恢复结果 |

#### 异常类型 4：严重错误回滚

| 项目 | 内容 |
|------|------|
| **触发条件** | 发现严重错误，影响数据完整性 |
| **回滚步骤** | a. 备份当前状态<br>b. 恢复到上一个正确版本<br>c. 分析问题原因<br>d. 修正后重新执行 |
| **记录要求** | 记录回滚原因、回滚版本、修正内容 |

## 🎯 核心原则

1. **对事不对人**：关注系统和流程，不指责个人
2. **追根究底**：连续追问 5 个为什么，找到根本原因
3. **系统思维**：从系统角度思考问题，而非局部优化
4. **数据驱动**：用数据和事实说话，避免主观臆断
5. **持续改进**：复盘是起点，不是终点
6. **知识沉淀**：将经验转化为可复用的知识
7. **闭环管理**：问题→分析→方案→固化→验证→改进，形成闭环

## 📚 应用示例

### 示例 1：周报汇总遗漏
- 问题：周报汇总遗漏美术组工作内容
- 根本原因：缺乏标准化流程和清单式工作方法
- 解决方案：创建 Skill 文档，建立标准流程，分块读取数据
- 预防机制：完整性验证清单，岗位平等原则

### 示例 2：消息重复发送
- 问题：飞书卡片消息重复发送 3 条
- 根本原因：未检查工具调用结果就重试
- 解决方案：建立发送状态追踪机制
- 预防机制：发送前检查清单，错误类型判断

### 示例 3：API Key 泄露
- 问题：代码中包含真实 API Key 并提交到 GitHub
- 根本原因：缺乏发布前检查流程
- 解决方案：建立打包前检查清单
- 预防机制：使用环境变量，自动化扫描

## 🔄 使用指南

**何时使用**：
- 发生工作失误时
- 发现流程缺陷时
- 系统出现故障时
- 客户投诉时
- 重复发生的问题

**使用步骤**：
1. 复制复盘报告模板
2. 按框架填写各部分
3. 组织相关人员讨论
4. 确定解决方案和责任人
5. 创建/更新 Skill 文档
6. 定期验证效果

**输出物**：
- 复盘报告（1 份）
- Skill 文档（1 份或多份）
- 检查清单（若干）
- 改进计划（1 份）
