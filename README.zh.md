## 💎 核心功能

### 谛听·识异（质量检测）

质量标准不是挂在墙上的标语，而是嵌入流程的门禁。

```
代码提交 → 自动检测 → 人工审查 → 质量门禁 → 允许发布
```

**检查清单** 已内置：
- ✅ 需求评审清单（5 项）
- ✅ 代码审查清单（5 项）
- ✅ 发布检查清单（5 项）

**质量报告** 自动生成：
```markdown
# 质量周报
- 代码覆盖率：85% (↑2%)
- 缺陷密度：0.5/KLOC (↓0.1)
- 发布成功率：100%
- 回滚率：0%
```

---

### 谛听·复盘（复盘分析）

不复盘，事故就白发生了。

**标准复盘流程**（4 步闭环）：

```
事故发生 → 信息收集 → 根因分析 → 改进措施 → 效果追踪
  ↓          ↓           ↓           ↓          ↓
24 小时内   48 小时内    72 小时内    持续      关闭
```

**核心指标** 自动计算：
| 指标 | 目标值 | 含义 |
|------|--------|------|
| MTTR | ≤30 分钟 | 平均恢复时间 |
| 复盘完成率 | 100% | 应复盘/已完成 |
| 改进完成率 | ≥90% | 计划改进/已完成 |
| 重复事故率 | ≤5% | 重复事故/总事故 |

---

### 谛听·图鉴（知识库）

让经验流动起来。

**问题库**：每个问题都是一张卡片
```
问题 ID → 识别特征 → 解决方案 → 预防措施
```

**检查清单库**：拿来即用
- ✅ 代码审查清单（基础/功能/性能/安全）
- ✅ 发布检查清单（发布前/中/后）
- ✅ 测试规范（单元/集成/性能）

**最佳实践库**：持续积累
- 代码规范（命名/注释/异常处理）
- 测试规范（覆盖率/边界/异常场景）
- 工具推荐（审查/分析/测试/压测）

### 核心教训（必须遵守）

> **LL-001：必须使用清单式工作**
> 永远使用检查清单，不依赖印象式工作

> **LL-002：必须按岗位逐一审查**
> 每个岗位、每个成员都必须被审查到，不能跳过

> **LL-003：必须分块读取避免数据截断**
> 大数据必须分块读取，不能一次性读取

> **LL-004：必须交叉验证完整性**
> 写完必须验证，交叉检查，不能直接提交

> **LL-005：岗位平等原则**
> 程序、策划、美术同等重要，不能偏废

## 💡 核心理念

### 1. 问题不是敌人，遗忘才是

事故本身不可怕，可怕的是**白发生了**。
每一次事故，都是系统改进的机会。

### 2. 质量不是检查出来的，是生产出来的

质量门禁不是"卡人"，而是**保护团队**。
标准前置，问题早发现，成本最低。

### 3. 经验不流动，就是浪费

一个人踩过的坑，全团队都应该知道。
一个人总结的经验，应该成为团队的财富。

### 4. 改进不追踪，就是空话

改进措施没有负责人、没有时间、没有验证，
等于**没改进**。
