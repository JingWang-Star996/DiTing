## 💎 核心功能

### 谛听·识异（质量检测）

质量标准不是挂在墙上的标语，而是嵌入流程的门禁。

**工作流程**：
```
代码提交 → 自动检测 → 人工审查 → 质量门禁 → 允许发布
```

**检测项目**：
- ✅ 代码规范检查
- ✅ 单元测试运行
- ✅ 代码覆盖率检查（≥80%）
- ✅ 安全扫描

**质量门禁**：所有检查通过 → 允许发布，任一失败 → 打回修正

**交付物**：
- 需求评审清单（5 项）
- 代码审查清单（5 项）
- 发布检查清单（5 项）
- 质量周报（自动生成）

---

### 谛听·复盘（复盘分析）

不复盘，事故就白发生了。复盘不追踪，改进就是空话。

**标准流程**（4 步闭环）：

```
事故发生
   ↓
【1. 信息收集】24 小时内
   → 时间线、影响范围、处理过程
   ↓
【2. 根因分析】48 小时内
   → 5 Why 分析、鱼骨图、直接/根本原因
   ↓
【3. 改进措施】72 小时内
   → 短期措施 + 长期措施 + 负责人 + 完成时间
   ↓
【4. 效果追踪】持续
   → 执行 → 验证 → 关闭
```

**核心指标**：
| 指标 | 目标值 | 含义 |
|------|--------|------|
| MTTR | ≤30 分钟 | 平均恢复时间 |
| 复盘完成率 | 100% | 应复盘/已完成 |
| 改进完成率 | ≥90% | 计划改进/已完成 |
| 重复事故率 | ≤5% | 重复事故/总事故 |

**交付物**：
- 事故等级判定（P0-P3）
- 时间线记录（精确到分钟）
- 5 Why 根因分析（追问到系统层面）
- 改进措施追踪（短期 + 长期）
- 经验教训总结（做得好 + 需改进）

---

### 谛听·图鉴（知识库）

让经验流动起来。一个人踩过的坑，全团队都应该知道。

**问题库**：每个问题都是一张卡片

```
问题 ID → 识别特征 → 解决方案 → 预防措施
```

**示例**：
```markdown
# 问题卡片：空指针异常

**问题 ID**: PROB-001
**分类**: 代码问题 → Bug

## 识别特征
- 错误信息：NullPointerException
- 触发条件：特定输入或状态

## 解决方案
1. 添加空值检查
2. 使用 Optional 包装
3. 初始化默认值

## 预防措施
- [ ] 代码审查时检查
- [ ] 静态分析工具检测
- [ ] 单元测试覆盖
```

**检查清单库**：拿来即用
- ✅ 代码审查清单（基础/功能/性能/安全）
- ✅ 发布检查清单（发布前/中/后）
- ✅ 测试规范（单元/集成/性能）

**最佳实践库**：持续积累
- 代码规范（命名/注释/异常处理）
- 测试规范（覆盖率/边界/异常场景）
- 工具推荐（审查/分析/测试/压测）

### 核心教训（必须遵守）

> **LL-001：必须使用清单式工作**
> 永远使用检查清单，不依赖印象式工作

> **LL-002：必须按岗位逐一审查**
> 每个岗位、每个成员都必须被审查到，不能跳过

> **LL-003：必须分块读取避免数据截断**
> 大数据必须分块读取，不能一次性读取

> **LL-004：必须交叉验证完整性**
> 写完必须验证，交叉检查，不能直接提交

> **LL-005：岗位平等原则**
> 程序、策划、美术同等重要，不能偏废

## 💡 核心理念

### 1. 问题不是敌人，遗忘才是

事故本身不可怕，可怕的是**白发生了**。
每一次事故，都是系统改进的机会。

### 2. 质量不是检查出来的，是生产出来的

质量门禁不是"卡人"，而是**保护团队**。
标准前置，问题早发现，成本最低。

### 3. 经验不流动，就是浪费

一个人踩过的坑，全团队都应该知道。
一个人总结的经验，应该成为团队的财富。

### 4. 改进不追踪，就是空话

改进措施没有负责人、没有时间、没有验证，
等于**没改进**。
